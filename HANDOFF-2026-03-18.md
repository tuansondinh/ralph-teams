## Important findings

### 1. Release notes wording is probably too Codex-specific

The `v1.0.26` GitHub release notes currently say:

`Codex sessions now explicitly get access to the Ralph package directory when needed.`

That line is likely stale or at least over-emphasized now.

Why:

- the more important fix is backend-agnostic:
  - Ralph now stages a runtime-local `rjq` at `.ralph-teams/bin/rjq`
  - sets `RALPH_RJQ_BIN` to that local file
  - prepends `.ralph-teams/bin` to `PATH`
- that should help any sandboxed backend, not just Codex

Follow-up:

- update the GitHub release notes to emphasize the generic runtime-local `rjq` fix
- consider removing the Codex-only wording entirely unless we confirm `--add-dir "$SCRIPT_DIR"` is still needed

### 2. Codex-specific `--add-dir "$SCRIPT_DIR"` may now be redundant

We added Codex launch changes so sessions explicitly include `--add-dir "$SCRIPT_DIR"`.

That was useful before the runtime-local `rjq` staging existed.

Now that `rjq` is copied into `.ralph-teams/bin` inside the project runtime, the Codex-specific package-dir add may no longer be necessary for the `rjq` issue itself.

Follow-up:

- test removing the Codex `--add-dir "$SCRIPT_DIR"` additions
- verify nested Codex builder / validator / merger flows still work
- if confirmed redundant, remove it in a cleanup patch

### 3. OpenCode showed a separate-looking permission symptom

Observed log:

```text
[EPIC-001] Now I'll update the state file to mark US-001 as passed:
[EPIC-001] ! permission requested: external_directory (/Users/sonwork/Workspace/ralph-teams-web/.ralph-teams/state/*); auto-rejecting
[EPIC-001] ✗ bash failed
[EPIC-001] Error: PermissionRejectedError({"_tag":"PermissionRejectedError"})
```

Then the retry succeeded.

Likely explanation:

- worktrees currently get `.ralph-teams` as a symlink back to the root runtime dir
- OpenCode appears to treat the resolved target path as an external directory
- so even though the agent is operating under the worktree path, the real target becomes:
  - `/Users/sonwork/Workspace/ralph-teams-web/.ralph-teams/state/...`
- that can trigger `external_directory` rejection in sandboxed runs

Important nuance:

- the retry succeeding suggests this is not a hard blocker every time
- but it is still a real footgun for OpenCode / sandboxed backends

Follow-up:

- reproduce with OpenCode in a minimal project
- inspect whether state-file updates should avoid the `.ralph-teams` symlink target entirely
- likely fix direction:
  - avoid symlink-based runtime access for state/plan files inside worktrees, or
  - provide backend prompts/paths that stay strictly inside the worktree without resolving to the shared root runtime dir
